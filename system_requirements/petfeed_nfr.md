# Нефункциональные требования (NFR) — PetFeed v1.0

---

## NFR-PERF-001: Время отклика API
**Описание:** Backend API должен обрабатывать запросы в пределах допустимого времени при штатной нагрузке.

**Критерии измерения:**
- `GET`-запросы (профиль, рацион из кэша): не более 300 мс при 500 одновременных пользователях
- `POST`-запросы (создание профиля, запись веса): не более 500 мс
- Расчёт рациона (без кэша): не более 1 сек
- Ответ AI-ассистента (DeepSeek API): не более 10 сек (AC-007)
- 95-й перцентиль времени ответа не превышает указанных значений

**Инструменты:** Apache JMeter, Locust, Prometheus + Grafana

**Приоритет:** Критический

**Обоснование:** Telegram-бот воспринимается как "зависший" при задержке более 3 сек — пользователи уходят

## NFR-PERF-002: Пропускная способность
**Описание:** Система должна выдерживать планируемую нагрузку на всех фазах роста.

**Критерии измерения:**
- MVP (3 мес): не менее 50 RPS без деградации
- Фаза 2 (6 мес): не менее 200 RPS
- Фаза 3 (12 мес): не менее 500 RPS
- CPU: не более 70% при пиковой нагрузке
- RAM: не более 1 ГБ для Backend API

**Инструменты:** Locust, Prometheus, Grafana

**Приоритет:** Высокий

**Обоснование:** Рост аудитории с 1k до 20k пользователей предусмотрен roadmap

---

## NFR-PERF-003: Точность напоминаний
**Описание:** Уведомления о кормлении должны доставляться точно в указанное время.

**Критерии измерения:**
- Отклонение от запланированного времени: не более ±1 минуты (AC-004)
- При недоступности Telegram API — повтор через 5 минут, максимум 3 попытки

**Инструменты:** Ручное тестирование, логи APScheduler

**Приоритет:** Высокий

**Обоснование:** Нарушение точности напоминаний — прямой удар по ключевой ценности продукта

## NFR-SEC-001: Аутентификация и авторизация
**Описание:** Все запросы к API должны быть аутентифицированы. Пользователь видит только свои данные.

**Критерии измерения:**
- Запрос без `X-Telegram-Id` возвращает `401 Unauthorized`
- Запрос чужого `pet_id` возвращает `404 Not Found` (не раскрывать существование ресурса)
- Webhook партнёра без валидного `X-Api-Key` возвращает `401 Unauthorized`
- Admin-эндпоинты доступны только с `X-Admin-Token`
- 100% эндпоинтов покрыты проверкой авторизации

**Инструменты:** OWASP ZAP, ручное тестирование

**Приоритет:** Критический

**Обоснование:** Утечка данных питомцев и истории заказов недопустима

---

## NFR-SEC-002: Защита данных
**Описание:** Персональные данные пользователей и чувствительная информация должны быть защищены.

**Критерии измерения:**
- Передача данных: TLS 1.2+ для всех соединений
- API-ключи партнёров хранятся только в виде bcrypt-хэша (cost factor ≥ 12)
- Admin-токен хранится только в переменных окружения, не в коде и не в БД
- `telegram_id` — единственный персональный идентификатор (ФИО не собирается)
- Логи не содержат персональных данных пользователей

**Инструменты:** SonarQube, ручной аудит

**Приоритет:** Критический

**Обоснование:** Соответствие минимальным требованиям GDPR / ФЗ-152

---

## NFR-SEC-003: Rate limiting
**Описание:** Защита от злоупотреблений и DDoS-атак на уровне API.

**Критерии измерения:**
- AI-ассистент: не более 10 запросов в день на пользователя (AC-007)
- Общий rate limit на IP: не более 100 запросов в минуту
- Webhook партнёра: не более 1000 запросов в минуту с одного IP
- При превышении лимита — `429 Too Many Requests` с заголовком `Retry-After`

**Инструменты:** FastAPI middleware, Redis (счётчики), OWASP ZAP

**Приоритет:** Высокий

**Обоснование:** Злоупотребление AI-запросами напрямую влияет на расходы на DeepSeek API

---

## NFR-REL-001: Доступность системы
**Описание:** Система должна быть доступна пользователям в любое время суток.

**Критерии измерения:**
- Доступность: не менее 99.5% в месяц (≤ 3.6 часа простоя)
- MTTR (среднее время восстановления): не более 30 минут
- MTBF (среднее время между сбоями): не менее 720 часов
- Плановые работы: только в 02:00–04:00 МСК с уведомлением пользователей

**Инструменты:** UptimeRobot, Grafana Alerting

**Приоритет:** Высокий

**Обоснование:** Животных кормят по расписанию — недоступность бота в момент напоминания критична

---

## NFR-REL-002: Устойчивость к отказу внешних сервисов
**Описание:** Отказ внешних API не должен приводить к полной неработоспособности системы.

**Критерии измерения:**
- При недоступности DeepSeek API: fallback на Redis-кэш, `503` если кэша нет — остальные функции работают
- При недоступности партнёра: отображать сообщение об ошибке, остальные функции работают
- При недоступности Redis: система продолжает работу через БД (деградация производительности допустима)
- Таймаут запроса к DeepSeek API: 15 секунд

**Инструменты:** Chaos Engineering (ручное), мониторинг Prometheus

**Приоритет:** Высокий

**Обоснование:** Деградация одного модуля не должна ронять весь бот

---

## NFR-SCAL-001: Горизонтальное масштабирование
**Описание:** Backend API должен поддерживать горизонтальное масштабирование без изменения кода.

**Критерии измерения:**
- Приложение stateless: состояние только в PostgreSQL и Redis, не в памяти процесса
- FSM-состояния диалога хранятся в Redis, не в памяти бота
- Масштабирование до 3 инстансов Backend API без деградации функционала
- Feature flags кэшируются в Redis с TTL 60 сек — инвалидация при изменении

**Инструменты:** Docker Compose (dev), Railway / VPS (prod)

**Приоритет:** Средний

**Обоснование:** Подготовка к росту аудитории с минимальными изменениями инфраструктуры

---

## NFR-USAB-001: Время онбординга
**Описание:** Новый пользователь должен получить первые рекомендации за минимальное время.

**Критерии измерения:**
- Создание профиля питомца: не более 7 шагов диалога (AC-001)
- Время от /start до первых рекомендаций: не более 3 минут
- Количество кнопок в одном сообщении: не более 6
- Все сообщения бота на русском языке

**Инструменты:** Ручное тестирование, замер времени сессии

**Приоритет:** Высокий

**Обоснование:** Высокий bounce rate на онбординге — главная угроза retention

---

## NFR-COMP-001: Совместимость с Telegram
**Описание:** Бот должен корректно работать на всех платформах Telegram.

**Критерии измерения:**
- Поддержка: Telegram iOS, Android, Web, Desktop
- Inline-кнопки работают корректно на всех платформах
- Сообщения не превышают лимит Telegram: 4096 символов
- Webhook обрабатывается в течение 60 секунд (лимит Telegram Bot API)

**Инструменты:** Ручное тестирование на каждой платформе

**Приоритет:** Критический

**Обоснование:** Telegram — единственный канал коммуникации с пользователем в MVP

---

## NFR-MAINT-001: Качество кода и тестирование
**Описание:** Кодовая база должна быть поддерживаемой и покрытой тестами.

**Критерии измерения:**
- Покрытие unit-тестами бизнес-логики (Service Layer): не менее 70%
- Покрытие интеграционными тестами ключевых эндпоинтов: не менее 50%
- Линтинг: код проходит проверку ruff без ошибок
- Время деплоя новой версии: не более 10 минут
- Все feature flags задокументированы в `feature_flags.md`

**Инструменты:** pytest, pytest-cov, ruff, GitHub Actions

**Приоритет:** Средний

**Обоснование:** Скорость итераций в стартапе критична — плохое покрытие замедляет разработку

---

## NFR-MAINT-002: Логирование и мониторинг
**Описание:** Система должна предоставлять достаточно информации для диагностики инцидентов.

**Критерии измерения:**
- Все запросы к API логируются: метод, путь, статус, время ответа
- Ошибки 500 логируются с полным stack trace
- Запросы к DeepSeek API логируются: время ответа, статус, количество токенов
- Логи retention: не менее 30 дней
- Алерт при ошибках 500 > 5 в минуту

**Инструменты:** Python logging, Grafana Loki (или Railway logs)

**Приоритет:** Высокий

**Обоснование:** Без логов невозможна быстрая диагностика — MTTR вырастет выше 30 минут
